\section{Test goals}
This section lays out the basic objectives of the tests and a simple overview of how they can be achieved.

\subsection{Photodetector}
%\subsection{\glsfmtshort{tia}}
There are two suitable methods of characterizing the \gls{tia}. As the main point of interest is the linearity of the amplifier, the first characterization method is to simulate photodiode readings at different light intensities by modeling it as a current source. By varying the diode model current, either by sweeps or manually, the performance of the physical device can be compared to the simulations.

The second method supplements the results of the first one. It involves measuring the scattering parameters $S_{11}$ and $S_{21}$ and calculating the frequency response of the system. Equation \eqref{eq:test:s11s22} shows the formula used to approximate the transimpedance $Z_t$ for a load resistor $R_l$ \cite{sackinger2017analysis}. This method provides a better overview of the transimpedance over the whole bandwidth of the photodetector.

\begin{equation}\label{eq:test:s11s22}
	Z_t(f) \approx \frac{S_{21}(f)R_l}{1 - S_{11}(f)}
\end{equation}

Neither of these methods uses a real photodiode, which is why an integration test is also necessary. Not only is the integration test needed to evaluate the performance of the diode, but it also gives insight into how the diode interacts with the laser, which will be used to conduct experiments with the sensing setup. Using the laser driver, a square-wave signal is pulsed at different frequencies within the amplifier bandwidth. By comparing the signal output by the photodetector to the signal output by the function generator, the performance of the system can be determined. Although the laser pulsing limits the test to square waves only, the filtering of high-frequency components at near-cutoff frequencies gives a deeper understanding of the behavior of the circuit.

\subsection{$T_1$ software}
It is important to reiterate, before delving into testing details, that as of completing this report, the hardware setup for $T_1$ measurements has still not been completed. This limit on the implementation and testing possibilities made it impossible to complete the software development and integration, which is why only certain unit tests could be done. Furthermore, testing was hampered by hardware unavailability and limited lab access.

Aside from proper functioning, the client set no requirements for speed. This is because the application relies on several fixed-length delay in order to operate. In addition, a longer runtime allows for more precise measurements. Because of this, all subsystems were tested and compared to the expected behavior with no regard for the length of the execution time.

\section{Test setup}
Because different test methods require slightly different physical setups, this section outlines the parts that comprise the setups and how they operate. Each setup is dissected separately and the necessary devices and components are shown in diagrams and pictures.

\subsection{Photodiode-model characterization}
Characterizing the system well requires fixed currents, due to the big difference in output voltage small currents differences at the input make. As the diode detects ambient light and cannot be set to one current level reliably, a model needs to be used instead. A voltage source can then be used to simulate the nanoampere photovoltaic current $I_f$ generated by the diode. According to Säckinger \cite{sackinger2017analysis}, the resistor $R_d$ and the capacitor $C_d$ in Figure \ref{fig:test:diode_equiv} are enough to characterize a \gls{tia} circuit. By inputting a voltage at the node $V_i$, $R_d$ functions as a current source, similar to a photodiode. This model assumes the parallel shunt resistance is so high that it can be omitted, while the series resistance can be neglected due to its limited impact on the system. However the diode capacitance is central to \gls{tia} analysis and design, which is why $C_d$ is there to mimic it. Additionally, there is the matching resistor $R_i$, which can be replaced by a coaxial terminator, and the capacitor $C_i$ that is used to isolate the input from the rest of the circuit.

Using this model, the diode current $I_f$ can be approximated using Equation \eqref{eq:test:i_f}. The approximation is true only if the input impedance $Z_i$ (see Equation \eqref{eq:test:z_i}) of the \gls{tia} is significantly lower than $R_d$ \cite{razavi2019transimpedance}. Keeping this in mind, a resistor six to seven orders of magnitude bigger was selected for the physical circuit.

\begin{equation}\label{eq:test:i_f}
	I_f = \frac{V_i}{R_d}
\end{equation}

\begin{equation}\label{eq:test:z_i}
	Z_i = \frac{R_1}{A_o + 1}
\end{equation}

For the tests, it was decided to test the \gls{dc} characteristics of the amplifier separately from the frequency response. Because of $C_i$, a \gls{dc} voltage will not generate a current. Instead, to verify that the \gls{dc} amplification matches the simulations, a low-frequency sine wave is used and its amplitude is varied, all while the output voltage is measured. The frequency response test measures the response of the system at near-cutoff frequencies, similar to the integration test. The difference is that instead of pulsing the laser, sine waves generate current using the photodiode model, which is then amplified. The change of waveforms is necessary because of the distorting effect of $C_i$ on square waves, especially at lower frequencies.

Figure \ref{fig:test:setupmodel} shows the assembled setup. A 1 \unit{\mega\ohm} resistor was used for $R_d$. Although the diode capacitance $C_d$ is very important for the frequency response, a 27 \unit{\pico\farad} capacitor was used instead of a 25 \unit{\pico\farad} one. This small of a difference should not affect the results much.

\subsection{Integration test}
The integration test has a similar test setup to the complete quantum sensing setup. However, the light from the laser directly illuminates the photodiode instead of a diamond sample \gls{nv}. This was done on purpose, because this test layout provides realistic information on the functioning of the photodetector when used in the quantum sensing setup without the added difficulty of processing the \gls{nv} signal. This test is particularly interesting for pulsed protocols, as a photodetector for them needs to be able to have a well-defined rising edge. A filter was placed in front of the photodetector to remove the green component of the laser and reduce the overall light intensity. However, some of the tests were done without filtering, as with manual adjustment the light intensity at the sensor can be set to an ideal level. When the tests were done, the new version of the laser driver was available and its \gls{pcb} included a laser, which is also shown in the diagram in Figure \ref{fig:test:tia_integ:setup}. For the \gls{ttl} pulses, a 5 \unit{\volt} signal is sent to the driver when in an on state.

The most interesting metric for data acquisition is the maximum sampling frequency before data loss occurs, as clear results require it to be much higher than the signal frequency. According to Nyquist's theorem, to sample a sinusoidal signal with a frequency of $f_{sig}$ the sampling frequency $f_s$ needs to be $f_s > 2f_{sig}$. However, square waves are comprised of multiple harmonics, in addition to the fundamental of the signal and this complex composition necessitates a drastic increase in $f_s$. In fact, to sample a usable amount of harmonics\footnote{Square waves contain so-called odd harmonics, which are harmonics at frequencies that are multiples of the fundamental and odd integer series. Minimum square-wave definition can be had with only up to the second harmonic, which has a frequency of $5f_{sig}$}, sampling should be done with a frequency $f_s > 2f_{harm2} = 10f_{sig}$. Lastly, because the measured signal is not periodic, the lowest pulse duration, which is the minimum dark time, needs to be taken to calculate $f_{sig}$. As a result the sampling frequency needs to be set to $f_s > \frac{10}{\tau_{dmin}}$.

\section{Integration test results}\label{chap:test:integ_res}
Tests were originally done with the laser placed less than 1 \unit{cm} away from the photodiode. However, better measurements can be achieved by moving the laser further away and not aligning the center of the beam with the diode. Doing both reduces the light intensity at the photodiode, which in turn prevents the sensor from being overexposed. In photovoltaic mode, overexposure severely lengthens the rise and fall time of the photodiode, and this effect is further exacerbated by the extremely high sensitivity of the circuit. Figure \ref{fig:test:v1_test} shows how the photodetector performs under close to ideal conditions. In these tests, the laser intensity should be barely enough for the signal to exhibit supply clipping. Lastly, the final \gls{cwodmr} and \gls{podmr} detectors were not tested for the reasons mentioned in Chapter \ref{chap:test:pdmodel}.

The most likely explanation for the bad results is the photodiode biasing. While in theory there is no voltage drop over a photodiode when in photovoltaic mode, in practice there are several millivolts of voltage across the diode, which still set the bias of the photodiode. According to Säckinger \cite{sackinger2017analysis}, this voltage greatly diminishes when a bootstrap is implemented and as a result the output current diminishes. This would explain the decrease in amplitude and the distorted shapes of the waveforms. 

To remedy the bias issue of the photodiode, a biasing circuit was implemented using an \gls{ac} coupling capacitor between the buffer output and the photodiode. When this did not work, one end of a resistor was connected to the anode of the diode and the other end was connected to a variable voltage source in order to test different bias voltages. The values that were tried ranged between -5 \unit{\milli\volt} and -5 \unit{\volt}, but none of them could bias the diode so that it would detect light. Instead, the signals shown in Figure \ref{fig:td:imp:v4biased} were output, no matter if the diode was illuminated or not. Changing the voltage of the source did affect the signal amplitude somewhat, but it remained mostly the same in shape. The low-frequency view of the noise, shown in Figure \ref{fig:td:v4biasedlowf}, exhibits an almost square-wave shape, but the high-frequency view in Figure \ref{fig:td:v4biasedhighf} shows a periodic pulse with a frequency of around 750 \unit{\kilo\hertz}.

\section{Discussion}
After presenting the results in the previous section, this section interprets the data and their meaning in a quantum sensing context.

\subsection{Photodetectors}
As a result of the tests, it was concluded that even the first iteration of the photodetector can be used for \gls{cwodmr} measurements. The protocol can be executed with extremely-low-bandwidth photodetectors (e.g. 18 \unit{\hertz} \cite{acharya2025compact}). Of course, later version present a better overall solution with less output noise. That being said, pulsed protocols cannot be achieved with the first iterations of the circuit due to their high frequency requirements. Higher frequencies also exacerbate the issue of overexposure of the sensor, which leads to decreased performance. The effect of the laser on the sensor needs to be considered when integrating the complete setup, as it can affect the measurements. There are cancellation techniques to measure only the \gls{nv} luminescence \cite{sewani2020coherent}, but if the photodiode is overexposed or the signal is getting limited by the power rail, the output will not be usable. 

Even though the first iteration has passable characteristics, it is worse than the second version in all regards. The improved \gls{tia} design presents a better frequency response. Across all frequencies, the device shows less attenuation of the harmonics of the square wave, as well as more defined rising and falling edges. In addition, there is less noise present in the signal. This can be attributed to the addition of the ground plane, as well as the better layout of the \gls{pcb}. 

The third iteration offers a solution that is more suitable for a compact setup. The frequency response is the same as the second version, but the onboard power supply inverter generates unwanted noise. As seen in Figure \ref{fig:test:v3ripple_both}, there is a large periodic signal, that looks like a supply ripple. It is also possible that the 1 \unit{\micro\farad} filtering capacitors $C_O$, $C_I$ and $C_{fly}$ do not match the design values, because they were found in a capacitor book in the FabLab. Due to the unknown manufacturer and model, the \gls{esr} of $C_O$ might also be too big. Furthermore, there is a possibility that their capacitance is lower than 1 \unit{\micro\farad}, which would also explain the shape of the ripple in Figure \ref{fig:test:v3ripple}. The presence of 50 \unit{\kilo\hertz} noise after increasing the capacitance, in combination with the large noise amplitude, suggests there is interference with the output of the amplifier.

The fourth iteration was shown to not function, even after modifying the system by adding a biasing circuit. While it is possible the issue was caused by faulty components, the op amps were changed with no effect, suggesting the biasing was indeed the culprit. However, this cannot be confirmed, as even after the addition of the biasing circuit, the \gls{tia} still does not work.

\subsection{$T_1$ software}
The software for generating pulsing sequences and doing $T_1$ measurements, or at least the subsystems that could be completed and integrated, works as intended. 

Generating pulsing sequences can be achieved with dark times in the required range of 200 \unit{\nano\second} to 2 \unit{\micro\second} with some slight signal distortion as the dark time decreases. Additionally, the part-linear, part-logarithmic function that is used for spacing the dark times, so that there are more data points at lower values of $\tau_{d}$, functions as expected, which was clearly demonstrated in Figure \ref{fig:td:logspace}.

However, the software is limited in what it can do in terms of data acquisition by hardware restrictions. There is data loss that can occur at the sampling frequencies required to measure $T_1$, as was shown in Chapter \ref{chap:test:daq}. The loss of data is entirely because the \gls{ad2}, which was used in the experiments, has problems keeping up with the high frequency demands of the $T_1$ protocol. What is more, the tests were done with the highest minimum dark time $\tau_{dmin}$ possible and lower $\tau_{dmin}$ values need to be achieved for the hardware setup to be viable.


\section{Known limitations}
The most obvious limitation is the lack of equipment for some tests, which lead to using less appropriate alternatives or having to cancel tests. For example, due to the lack of a suitable \gls{vna}, the only method for circuit characterization was using a diode model. The latter would also benefit from a real nanoampere current source instead of the makeshift source that had to be used. The diode model using the 1 \unit{\mega\ohm} resistor also contributed significant, which was amplified by the \gls{tia}.

There are many factors that affect the measurements, but perhaps most prominent are the environmental lighting sources, at least when it comes to the integration tests. This includes ambient sunlight, which gets through the blinds of the lab, and reflections from the laser. Both significantly affect the photodetector, especially when there is no focused light present. Reflections, in particular, majorly contributed to signal distortion at higher frequencies.

Furthermore, the light source in the medium-distance integration tests had to be manually adjusted, meaning that the output signals do not represent the ideal performance. In fact, the measurements present results with slight power rail clipping, which is unavoidable with the testing setup. To achieve ideal readings, the laser would have to be affixed and the orientation adjusted, while the intensity at the given distance is measured.

Lastly, the software testing was hindered severely by hardware availability. At the time of testing, the \gls{ad2}, as well as the external oscilloscope that was used for verification, had to be shared with other students and employees of the research group. Additionally, the lab access was restricted, as there was a holiday, during which the lab was locked. Lab access was limited even further by an administrative misunderstanding. Both of these circumstances led to testing that was not extensive as originally planned. Most subsystems were tested and verified to work, but edge cases might have been missed, owing to the aforementioned external circumstances. The \gls{daq} tests could have also been more detailed, if the \gls{ad2} was available more often.


\chapter{Conclusion}
In conclusion, the main goal of the project, to enable detection of light signals for quantum sensing, was achieved. While the feasibility of an inexpensive photodetector for pulsed protocols has yet to be verified in practice, the proposed design presented in this paper can be used as a reference for future implementation. All designs adhere to the size requirements and cost significantly less than the 50 \texteuro budget per board (around 30 \texteuro). The second most important goal of creating a software for signal processing and visualization was also mostly completed. It can do all of the steps that are required for measuring $T_1$, except for decoding the acquired signals. All future work that is needed to make the software fully-functional is outlined in Chapter \ref{chap:recs}.

According to the introduction in Chapter \ref{chap:intro}, the main purpose of the project was create a photodetector, but integration of the sensing setup was also important. Both were tackled during this project, with the focus being on photodetection. While setup integration also holds importance for the research group, most of the tasks related to it were redistributed among other students. Further refinements to the photodetector and the setup as a whole are possible. Some of the most promising ones are discussed in the recommendations in Chapter \ref{chap:recs}. Complete integration of the whole setup necessitates more work, but is achievable in the near future.

The first iteration of the photodetector is suitable for \gls{cwodmr} measurements with the quantum sensing setup. Despite its functionality, it also presents some minor integration problems, such the need for a dual-rail power supply. 

Perhaps most suitable of all for low-noise \gls{cwodmr} measurements, the second version showed promising results and is most suited for a lab setup. Out of all improvements, the noise reduction is the most significant. However, the need for a dual-rail power supply limits the usability in more compact and self-contained future implementations.

Version three of the device addresses the major integration concern of having a simple and small power supply solution. At the same time, it also incorporates the improved system response of the second iteration. Although the \gls{ac} characteristics of the circuit are irrelevant for \gls{cwodmr}, they become the most important aspect of the circuit when discussing pulsed protocols. However, the third version of the photodetector suffers a significant noise problem, caused by the inverting charge pump. Although this is a significant problem, increasing the filtering capacitors showed a major improvement. Even if this version is not recommended for a desktop setup, microcontroller-based portable solutions will benefit from the simplified power supply and will not need an external module to supply -5 \unit{\volt} to the detector.

The fourth photodetector iteration was an attempt at a pulsed protocol solution. At the very least, the version was supposed to handle the $T_1$ measurements protocol, as it has the most attainable specifications of all pulsed protocols. Unfortunately, due to a biasing issue, the photodetector did not amplify the signal nearly enough. Two separate biasing circuits were also tried, but both resulted in only noise being output. Furthermore, there is a significant amount of noise present in the output signal, an issue which might prevent the sensor from measuring $T_1$ even if there was no biasing issue.

Lastly, the software application for doing $T_1$ measurements is mostly completed, with only the signal decoder missing, due to the absence of a finished physical setup to test and verify with. The rest of the code can be used in its current state, but there are some improvements, discussed in Chapter \ref{chap:recs}, that can improve the efficiency and stability of the measurements.

\section{Photodetector}\label{chap:recs:photodetector}
There are several recommendations that are important for future implementations, as they may contribute to the precision, noise-suppression and stability of the photodetector and the quantum sensing setup as a whole. 


\begin{figure}[!ht]

Firstly, increasing the photodetector bandwidth should be the main priority in the future, as no version of the detector can measure pulsed protocols. By implementing the last \gls{podmr} iteration, even Rabi oscillations detection will be possible. The previous photodetector versions are bottlenecked by the low photocurrent generated by the photovolaic-mode diode. However, biasing a BPW34 in photoconductive mode will result in an increase in dark current, usually around \num{2} \unit{\nano\ampere}, but it can rise up to 30 \unit{\nano\ampere}, rendering the actual signal unreadable. Alternatively, the bandwidth can be increased by finding an op-amp with bigger \gls{gbp}. Out of all op-amps that were explored, the OPA818 shows the most promise, which is why its mathematical model was used to compare it to the AD795 and ADA4637. Figure \ref{fig:recs:opa818comparison} showcases the significant bandwidth increase from the ADA4637 to the OPA818. Using the former results in a \num{138,87} \unit{\kilo\hertz} bandwidth, while the latter results in a \num{982,44} \unit{\kilo\hertz} bandwidth. As is usually the case with bigger-\gls{gbp} op-amps, the OPA818 has worse noise parameters (25 \unit[power-half-as-sqrt]{\nano\volt\per\hertz\tothe{0.5}} at 100 \unit{\hertz} as compared to 7 \unit[power-half-as-sqrt]{\nano\volt\per\hertz\tothe{0.5}} at 100 \unit{\hertz}).

In the future, if even greater bandwidth is desired, transistor \glspl{tia} should be investigated. Transistors offer finer control over the parameters of the amplifier they constitute. This is, in part, due to the control over the technology of each transistor, but also because they are lower-level components than op-amps, which allows for more intricate layouts. Several sources on the matter were consulted \cite{sackinger2017analysis, santiago2021a, karimi2017silicon, analui2004bandwidth}, but because of time constraints and hardware availability, an op-amp-based solution was chosen instead.

Secondly, exploring avalanche photodiodes as an alternative to the Si diode (BPW34) currently in use, might contribute to better measurements. Although they are widely used in low-intensity applications, avalanche diodes are quite expensive (upwards of 100 \texteuro). The client wanted to keep the cost low for this project, but in the future more precise pulsed protocol measurements might benefit from the internal amplification and increased precision that are offered by avalanche photodiodes. There are also p-i-n alternatives that might provide better performance than the BPW34. InGaAs diodes are often used in telecommunication due to their stable, low-noise performance even at high frequencies. Their peak sensitivity is also usually in the red and infrared area of the light spectrum, which means the noise resulting from the green laser will be proportionally smaller than with the BPW34. Similar to avalanche photodiodes, the major drawback of InGaAs diodes is their price, which is much bigger than that of the BPW34 and similar to that of avalanche photodiodes. Some of the more expensive diode alternatives (InGaAs and avalanche) that were explored as possible replacements for the BPW34 were the Hamamatsu G6854-01, Hamamatsu G17190-003K, Luna PDU-V104, OSI FCI-InGaAs-1000 and Marktech MTAPD-06-009. Some of the better affordable Si alternatives are the OSRAM SFH 203 FA and Vishay TEMD1000, which detect less green light, have a lower dark current and have better, more narrow, directional characteristics, possibly leading to less ambient light noise. However, they also have smaller photosensitive areas, which result in less photocurrent, but also in smaller junction capacitances \cite{hm2025si, lt2006area}. This drastic reduction in input capacitance, leads to significant increase in bandwidth (see Figure \ref{fig:recs:diodecomparison}) as it necessitates smaller compensation capacitors for the same frequency response. Ultimately, there is no single best option for the setup, as all have their advantages and downsides. However, for the current price requirements, the TEMD1000 is the best photodiode and because of its characteristics it was used in the proposed \gls{podmr}[] detector design. 


Thirdly, its possible to increase the \gls{snr} of the \gls{cwodmr} sensor by implementing a balanced photodetector \cite{Paschotta_2019_balanced_photodetection}. The difference between the basic \gls{tia}[] circuit shown in Figure \ref{fig:tia} and a balanced photodetector \gls{tia} is that the latter has a second photodiode connected in series with the first one, so that what is amplified by the \gls{tia} is the difference between their signals. By sending a signal containing the fluorescence and a reference laser signal, the fluorescent signal can be isolated for the most part. This addition reduces the optical noise contributed by the fluctuations of the green laser significantly and can increase the magnetic sensitivity of the sensing setup by several orders of magnitude \cite{li2023noise}. For these reasons, such an implementation was considered, however the added complexity to the optical data path dissuaded the client from pursuing the idea further, as building towards the Rabi oscillations protocol is more important than refining \gls{cwodmr} for the time being.

\section{Software application}
To fully complete the $T_1$ measurement application, a data decoder needs to be implemented, which is possible only after the physical sensing platform has been finished and $T_1$ measurements have been decoded manually. Given a file of unknown length, containing a number of $T_1$ measurements, the algorithm needs to start by detecting the edges of the pulses, based on which the discrete time duration of the measurements data needs to be turned back to continuous time. The most simple way of implementing this is by relating the continuous-time laser pulse duration entered by the user $\tau_{las}$ to the detected pulse. Afterwards, the different pulses have to be enumerated or made distinguishable in a different way so that every readout pulse can be used for the measurements, while the initialization and cancellation pulses are skipped. When a readout pulse is processed, the $T_1$ signal will be superimposed on the laser pulse. It is difficult to say how the extraction of the signal will be done with no setup to measure and experiment with, but there should be a transient with an exponential decay shape present in the signal and determining the time it takes for the transient to reach the laser signal level should give the time for the data point. Doing this for all measurements in the file and then returning the average should provide the time value for the given data point. Collecting and compiling the processed data from all data points should result in a plot, similar to Figure \ref{fig:t1result}, which was presented when first introducing the $T_1$ protocol.

Finalizing the application only requires signal decoding, but there are some functionalities that can be improved to make the measurements more stable. Firstly, implementing adaptive sampling frequency will reduce the size of the measurement files, which currently take up around 250 \unit{\mega\byte} of memory per data point. The latest version of the code at the time of finalizing the report has a fixed sampling frequency $f_s$ of 5 \unit{\mega\hertz}. In Chapter \ref{chap:test:daq}, the minimum value of $f_s$ was determined to be 5 \unit{\mega\hertz}, but because of the limit imposed on it by the 10 \unit{\mega\hertz} clock, it was decided that the sampling frequency should not be kept fixed. However, as $f_s$ based on the minimum dark time, data points with lower frequency requirements will be oversampled. Adapting $f_s$ for each data point, by decreasing it whenever necessary, will result in smaller file sizes and less data loss. It will, however, also contribute to a somewhat more complex decoding process and can only be implemented when, through manual measurements, the $f_s$ required for $T_1$ has been established. Secondly, the method of data acquisition can be changed from data stream acquisition to a number of consecutive single buffer acquisitions. Implementing this method of acquisition will be much more complicated, but it might entirely eliminate the data loss problem. The complexity comes from the triggering that will need to be done for the multi-buffer acquisition to be timed correctly. \lstinline|AnalogIn_Buffers.py|, which is an example file that comes with the \gls{sdk} download, provides an overview of how such an acquisition can be set up. Thirdly, a more complex device management system can be implemented to handle disconnection during tests. The Waveforms \gls{api} provides device enumeration functions, that all start with \lstinline|FDwfEnum|, capable of indexing all connected devices and returning their status. Monitoring disconnections will be helpful for informing the user, but it will also require another threaded function to call all necessary \gls{api} functions.

Perhaps most important of all considerations, pertaining to the $T_1$ measurements application, is what hardware is used for sequence generation and data acquisition. It has already been shown that the \gls{ad2} is the biggest bottleneck of the system, specifically when it comes to data acquisition. With the current $T_1$ protocol timings, the frequency demands cannot be met by the oscilloscope of the device. An alternative that has been thoroughly discussed with the client is the Analog Discovery 3. Its specifications are somewhat better than its predecessor, but the \gls{ad2} is also supposed to perform much better than it actually does. That being said, the oscilloscope of the newer device should support triple the sampling frequency and, as a consequence, signals with a dark time of as little as \num{666,67} \unit{\nano\second} should be measurable. Alternatively, the lab oscilloscope, a Rigol MSO1074Z, can be used for data acquisition. The device has much better specifications and should support a tenfold increase (dark time of \num{66,67} \unit{\nano\second}) in sampling frequency, at the very minimum. It is possible the increase is even greater, as the Analog Discovery setup uses the device to read data while simultaneously generating signals, thus using more resources and likely increasing the amount of data loss. That being said, the Rigol oscilloscope will require the use of an additional \gls{api}. Unfortunately, this Rigol series does not have a proprietary \gls{api} and, based on internet forum research, third-party libraries can be used with varying degrees of success. Another possible alternative to the \gls{ad2} is a \gls{fpga} board like the Red Pitaya STEMlab 125-14, which the research group already owns. It can measure signals 5 times faster the \gls{ad2}, but, just like the \gls{ad2}, it has the downside of only having 2 output ports, making Rabi oscillation support impossible without using two units of the same device.



\section{Lock-in amplification}
Data acquisition, or at least the hardware needed for it, is the main photodetector bottleneck. While previous sections discussed the \gls{ad2} as the main platform for data acquisition, \glspl{lia} are more commonly used in quantum sensing. Using a \gls{lia} was considered as an option, but the client wanted to explore the possibility of handling all inputs and outputs with the \gls{ad2}. Originally, a \gls{lia} was supposed to be used in tandem with the \gls{ad2}, as shown in Figure \ref{fig:td:ad2}, but the limited sampling frequency of the \gls{ad2} will result in data loss. An alternative solution is to remove the \gls{ad2} from the output data path. Instead of the \gls{ad2} mediating the hardware/software connection, the \gls{lia} can interface directly with the application. The HF2LI amplifier, which is the main \gls{lia}[] that has been explored during this project, has a 50 \unit[]{\mega\hertz} central frequency range, which should be more than sufficient for implementing $T_1$ measurements and Rabi oscillations.

As previously mentioned, using the \gls{lia} as an output device necessitates the replacement of the \gls{ad2} data acquisition program module. The software used by the HF2LI is called LabOne and offers \gls{api} support for multiple languages, including Python. A Python package by the name of \lstinline|zhinst-core| can be used to implement \gls{daq} functionalities, although a newer, more high-level version called \lstinline|zhinst-toolkit| exists. The latter encapsulates the low-level functions of the former, which is why implementing the \gls{daq} module using \lstinline|zhinst-toolkit| is recommended, unless some of the more specific functions of \lstinline|zhinst-core| are required.

Zurich Instrument devices need data servers to operate, which are programs that facilitate communication between software and hardware. All devices, except for the HF2LI use a generic server, but because the \gls{lia} is old, it requires a specific instance of what is called an \lstinline|hf2| server. To connect to the HF2LI, a session object needs to be created and provided with the host address (usually \lstinline|localhost|). If not provided with any other arguments, the session instance will get assigned a generic or an \lstinline|hf2| server depending on the connected device. However, specifying the server type when known is preferred. Figure \ref{fig:rec:server} shows how the \gls{lia} with ID \lstinline|DEVXXXX| can be connected.


Reading and writing using the LabOne \gls{api} happens using a structure called a node tree. It dynamically provides nodes that correspond to device subsystems, thus facilitating a quick and simple connection to a device. Reading data from the \gls{lia} should be done with a subscription to \lstinline|demods|, because it is the node containing the processed signal samples. Figure \ref{fig:rec:demods} demonstrates how, after data is extracted from the node, the data buffer containing all node information is polled, returning a dictionary with data from all active nodes. For more information about extracting data and writing to the \gls{lia}, consult the documentation \cite{zh_docs_node_tree}.
